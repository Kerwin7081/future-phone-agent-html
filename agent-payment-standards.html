Core Thesis

Agent 支付不是单一协议竞争，而是授权、身份、钱包、结算和责任归属的系统性重构。

Stripe 有一套，Coinbase 有一套，但行业并不止这两套。Google AP2、Visa TAP、Mastercard Agent Pay、PayPal 钱包层、Circle 稳定币结算层，以及 AWS 等云平台分发层，正在共同争夺 Agent 经济的支付基础设施位置。

Framework

Agent 支付可以拆成五层

01

购物与下单协议

解决 AI Agent 如何代表用户向商户发现商品、确认订单、传递购物意图。典型路线是 Stripe / OpenAI ACP。

02

支付授权协议

解决用户如何授权 Agent 花钱，金额、商户、时间、任务范围如何被绑定。典型路线是 Google AP2。

03

钱包与资金账户

解决 Agent 使用银行卡、PayPal、Link、稳定币钱包还是企业资金账户。这里是 Stripe、PayPal、Coinbase 的交叉地带。

04

结算网络

解决最终资金走卡网络、ACH、PayPal、USDC、PYUSD、银行间网络还是稳定币网络。Circle 与卡组织都在这一层竞争。

05

身份与责任归属

解决商户如何识别可信 Agent，支付出错、重复下单、退款和争议处理由谁承担。

AI

核心变化

Agent 从信息工具升级为经济执行主体，支付系统必须处理代理行为的可信性、可审计性和可追责性。

Standards Landscape

七条主要路线

S
Stripe / OpenAI ACP：AI 购物协议
ACP 解决的是 AI Agent 如何在 ChatGPT、AI App 和商户后端之间完成一次标准化购买流程。它适合电商、平台、SaaS 商户和 AI App 内购买。Stripe 的优势在于商户密度、开发者 API、Link 钱包、Checkout、Billing、Connect 和支付编排能力。
C
Coinbase x402：机器支付协议
x402 更接近 Agent 原生支付。它适合 API paywall、MCP server、数据源访问、AI 推理服务、模型调用、机器间结算和微支付。Coinbase 的优势在于 Base、USDC、Agent Wallets 和开发者生态。
G
Google AP2：授权与意图协议
AP2 的核心是 mandate，即用户授权 Agent 花钱的范围、金额、商户、时间和上下文。它不一定控制资金流，但可能成为跨平台 Agent 支付的授权中间层。
V
Visa TAP：可信 Agent 身份协议
Visa 关注的是商户如何识别可信 Agent，如何区分真实购买代理与异常自动化流量。它的重点是把 Agent 交易接入 Visa 的身份、风控和网络治理体系。
M
Mastercard Agent Pay：卡网络 Agent 支付基础设施
Mastercard 强调 tokenization、Know Your Agent 和现有卡网络治理。它的目标是让 Agent 成为卡网络上的新型支付主体，而不是让卡网络被绕开。
P
PayPal：钱包入口与争议处理层
PayPal 的优势是用户钱包、商户接受网络、买卖双方保护、物流追踪、退款和争议处理。Agent 购物越普及，用户越需要可信的钱包入口和售后责任机制。
U
Circle USDC / CPN / Gateway：稳定币结算层
Circle 不一定控制 Agent 购物入口，但 USDC、CPN、Managed Payments、Gateway 和 Nanopayments 可能成为 Agent 经济的资金结算层，尤其适合 24/7、跨境、可编程、小额和高频结算。
Route Comparison

几套标准的本质差异

ACP：解决购物流程

核心问题是 Agent 如何替用户在商户处下单。它最适合电商、品牌、平台、SaaS 商户、AI App 内购买和面向消费者的即时结账。

x402：解决资源付费

核心问题是 Agent 调用 API、数据、模型或工具时如何自动付费。它更像机器对机器支付协议，适合微支付和按次计费。

AP2：解决授权委托

核心问题是用户支付意图如何被验证、约束、撤销和审计。它可能成为不同钱包、商户和 Agent 平台之间的授权语言。

TAP / Agent Pay：解决可信身份

核心问题是商户和支付网络如何识别可信 Agent，并把 token、身份、风控和争议处理嵌入现有网络。

PayPal：解决用户信任

核心问题是用户愿意授权哪个钱包给 Agent 使用，以及错误订单、退款和争议如何处理。

Circle：解决最终结算

核心问题是高频、小额、跨境和 B2B Agent 支付最终使用什么资产和网络完成清算。

Adoption Path

未来不会只剩一个标准。

更可能的路径是：前端协议多样化，授权层逐渐标准化，底层结算多轨并存。消费者购物仍会大量依赖卡网络、钱包和退款保护。机器对机器支付、API 调用、MCP 工具和纳米支付，更可能使用稳定币和链上结算。企业级后台结算，则可能走 CPN、银行网络和托管稳定币结算。

Use Cases

不同场景会选择不同支付栈

AI 购物

ChatGPT、Gemini 或其他 AI App 帮用户买商品。核心需要 ACP、AP2、钱包、卡网络、退款与争议处理。

API 与 MCP 工具

Agent 调用数据源、搜索工具、推理服务或企业软件接口。核心需要 x402、稳定币钱包、按次计费和实时结算。

企业采购

企业 Agent 采购云资源、软件、机票、办公服务。核心需要 AP2、企业审批流、预算控制、审计和卡网络 token。

跨境结算

银行、PSP、Fintech 和企业财资部门做后台资金调拨。核心需要 CPN、稳定币托管结算、合规报表和流动性管理。

机器间协作

一个 Agent 调用另一个 Agent、模型或机器人服务。核心需要机器身份、自动授权、小额支付和可追踪凭证。

内容与数据付费

网页、数据库、研究内容和实时数据按次开放。核心需要低成本微支付、访问控制和使用记录。

Investment Lens

投顾视角：Agent 支付的四类资产

第一类：Agent 支付入口

代表：OpenAI、Stripe、Google、PayPal。核心看谁掌握用户和商户交互界面，谁能成为 Agent 默认支付入口。

第二类：Agent 钱包与协议

代表：Coinbase、Stripe、PayPal、MetaMask、Privy、Bridge 相关生态。核心看谁能成为 Agent 默认钱包和机器支付协议。

第三类：结算资产与网络

代表：Circle、Visa、Mastercard、PayPal、Coinbase。核心看稳定币、卡网络和钱包网络谁掌握最终清算层。

第四类：云平台分发层

代表：AWS、Google Cloud、Microsoft Azure。大量企业 Agent 会部署在云平台上，支付能力可能被直接嵌入 Agent 开发框架。

Company Read

上市资产映射

COIN

x402、Base、Agent Wallets 和 USDC 生态，使 Coinbase 具备较强的 Agent 原生支付协议属性。它的估值叙事有机会从交易所扩展到钱包、协议和机器支付基础设施。

Circle

USDC、CPN、Managed Payments、Gateway 和 Nanopayments，使 Circle 更接近稳定币结算资产和网络层。关键是非利息收入能否规模化。

Visa / Mastercard

两家公司不会轻易被稳定币绕开。它们的身份、token、风控、争议处理和全球商户接受网络，仍然是消费者 Agent 支付的重要基础设施。

PayPal

PayPal 的钱包、商户网络、买卖双方保护和争议处理能力，在 Agent 购物场景中可能重新获得战略价值。

Shopify

如果 AI 购物与商户后台深度连接，Shopify 生态商户可能成为 ACP 等协议的重要落地场景。

云平台

AWS Bedrock AgentCore Payments 表明，云平台可能把支付能力直接嵌入企业 Agent 工作流，成为新的分发层。

Risk Map

需要持续验证的风险

标准碎片化

协议太多可能导致商户、钱包、云平台和 Agent 开发者重复适配，短期降低采用效率。

责任边界不清

Agent 买错、重复执行、超出授权、订单争议等问题，必须明确用户、Agent 平台、钱包、商户和支付网络的责任划分。

监管差异

稳定币、消费者保护、数据权限、跨境资金流和身份认证在不同司法区差异较大。

商业化验证

短期产品发布不等于收入放量。要看真实交易量、take rate、客户留存和生态集成深度。

Conclusion

一句话总结

Stripe / OpenAI ACP 是 AI 购物协议。Coinbase x402 是机器微支付协议。Google AP2 是 Agent 授权协议。Visa TAP 是可信 Agent 身份协议。Mastercard Agent Pay 是卡网络 Agent 支付基础设施。PayPal 是钱包、商户保护和争议处理层。Circle 是稳定币结算网络和微支付资金层。

真正的大机会在于：Agent 从信息工具变成经济主体之后，谁负责授权、付款、结算、风控和责任归属。
